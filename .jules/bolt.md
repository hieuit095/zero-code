## 2024-04-03 - Agent Chat Re-render Bottleneck
**Learning:** In `AgentChatter.tsx`, high-frequency WebSocket streaming events (e.g., token-by-token LLM output) caused the entire message history to re-render. Because the `messages` array was mapped inline, every incoming token triggered a full DOM reconciliation for all historical chat bubbles, resulting in an O(N) render cost that degraded performance linearly as the chat lengthened.
**Action:** Always extract items mapped in high-frequency update loops (like streaming logs or chat tokens) into separate components wrapped in `React.memo()`. This creates an O(1) rendering cost where only the actively changing item re-renders, preventing UI stuttering and wasted CPU cycles.

## 2024-04-12 - Inline Mapping Performance
**Learning:** In `TasksPanel.tsx`, high-frequency state changes for single items (like the `qaRetryState` or `expanded` state toggled via `toggle`) caused the entire `tasks` array to re-render. Because the `tasks` array was mapped inline and the `toggle` function was recreated on every render, every minor state update triggered a full DOM reconciliation for all historical task bubbles, resulting in an O(N) render cost that degraded performance linearly as the list lengthened.
**Action:** Always extract items mapped in high-frequency update loops (like task lists) into separate components wrapped in `React.memo()`. Pair this by wrapping any passed functions (like `toggle`) in `useCallback` with empty dependencies (using functional state updates). This creates an O(1) rendering cost where only the actively changing item re-renders.
