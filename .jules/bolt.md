## 2024-04-03 - Agent Chat Re-render Bottleneck
**Learning:** In `AgentChatter.tsx`, high-frequency WebSocket streaming events (e.g., token-by-token LLM output) caused the entire message history to re-render. Because the `messages` array was mapped inline, every incoming token triggered a full DOM reconciliation for all historical chat bubbles, resulting in an O(N) render cost that degraded performance linearly as the chat lengthened.
**Action:** Always extract items mapped in high-frequency update loops (like streaming logs or chat tokens) into separate components wrapped in `React.memo()`. This creates an O(1) rendering cost where only the actively changing item re-renders, preventing UI stuttering and wasted CPU cycles.
## 2024-04-22 - LeftSidebar Recursive File Tree Re-render Bottleneck
**Learning:** In layout components like `LeftSidebar`, high-frequency update loops from sibling components (e.g., WebSocket streaming tokens from `AgentChatter` consuming `useAgentConnection`) cause the parent to re-render. If un-related expensive child components, such as the recursive `FileExplorer`, are not properly memoized, they inherit the re-render. In the case of `FileExplorer`, the inline `onSelect` callback and un-memoized `FileRow` components resulted in an O(N) cross-component re-render cascade for the entire file tree on every incoming token.
**Action:** When a parent layout component fetches high-frequency updates, ensure computationally expensive or pure child trees (like recursive file systems) are explicitly wrapped in `React.memo()` using an un-shadowed internal name (e.g. `const Comp = memo(function CompImpl() {})`). Always extract inline callbacks into stable `useCallback` hooks before passing them to these memoized children to successfully convert O(N) re-renders to O(1).
